# 法律咨询机器人 — 架构可行性分析

## 总体评估

这套架构**基本可行**，但距离真正的生产级还有明显的 gap。
下面按严重程度排序。

---

## 🚨 致命缺失（不上线会出事）

### 1. Human-in-the-Loop（人工审核）

**现状**：用户问 → AI 答，直接返回。

**问题**：法律咨询在中国属于特许经营行业，AI 直接出回答有合规风险。
而且 LLM 幻觉在法律场景完全不可接受——引用一条不存在的法条就是事故。

**建议**：加一个审核工作流

```
用户提问 → AI 生成草稿 → [lawyer 角色审核] → 用户看到最终回答
                        ↘ 驳回/修改 → 律师编辑后发布
```

审核流程需要的表：
- `consultations` — 咨询单（id, user_id, status: draft/published/rejected）
- `consultation_reviews` — 审核记录（consultation_id, reviewer_id, action, comment）

### 2. 法律合规（免责与声明）

- 每个回答末尾**强制拼接**免责声明："以上内容仅供参考，不构成法律意见。如涉及重大权益，请咨询执业律师。"
- 用户注册时需同意"AI 法律咨询须知"
- 敏感话题（刑事辩护、行政诉讼等）需要明确的转人工提示
- 数据合规：《个人信息保护法》要求用户数据加密存储、可删除

### 3. 知识库版本管理与准确性

法律是会修订的（民法典 2021 施行，刑法修正案十二 2024 施行）。
不能用"把 PDF 塞进去就完事"的方式。

**建议**：
- 每条法条增加 `effective_date`, `status(active/repealed/amended)` 字段
- 入库时按"效力级别"标注：法律 > 行政法规 > 司法解释 > 指导案例
- 向量库按版本隔离，法律更新时重建索引

---

## 🔶 架构缺失（建议补齐再上线）

### 4. 多轮对话管理

现在设计的是单轮 Q&A。实际用户会追问：

> 用户："租房合同纠纷怎么处理？"
> AI："根据民法典第 703 条..."
> 用户："那押金不退呢？"

需要：
- `sessions` 表（session_id, user_id, title, created_at）
- `messages` 表（session_id, role: user/assistant, content, sources, tokens）
- 上下文管理（窗口滑动 + token 计数）

### 5. 搜索质量：混合检索

只有向量检索在中文法律场景下不够。

- 法律术语多义性："拘留"在治安处罚和刑事中含义不同
- 用户不会用专业术语提问："被打了怎么办" vs "故意伤害罪"

**建议**：Qdrant 支持混合搜索（BM25 + 向量），打开即可。

### 6. LLM 响应缓存

法律问答有很多重复问题：
- "离婚冷静期是多久？" → 每天被问几十次
- 每次调用 DeepSeek API 都要花钱（~¥1-2/百万 token）

**建议**：加语义缓存（semantic cache）
- 新问题 → 计算 embedding → 在 Qdrant 中找语义相似的历史问答
- 相似度 > 0.95 → 直接返回缓存结果
- 缓存命中可以节省 30-50% 的 API 成本

### 7. 成本控制与配额

DeepSeek API 调用不是免费的：
- 需要给不同角色设置配额（user 每天 20 次、lawyer 不限）
- 需要监控每个用户的 Token 消耗
- 需要有告警机制（月消耗超过 ¥XXXX 时通知）

## 🟢 现有架构合理的地方

| 模块 | 评价 |
|---|---|
| FastAPI + SQLAlchemy async | 正确选择，异步性能好 |
| JWT + RBAC | 够用，不多不少 |
| Qdrant | 比 Milvus 轻量，比 Pinecone 可控，适合 |
| bge-large-zh-v1.5 | 中文法律领域最佳开源嵌入之一 |
| Docker Compose | 开发够用，生产建议加 k8s 或 swarm |
| LangSmith | 调试 Agent 必备 |
| ParentDocumentRetriever | 法律场景需要保留上下文，选对了 |

## 结论

**技术上可行**，核心架构选型合理。

但"生产级"的关键不在技术栈，而在**流程和合规**：
1. 最缺的是**审核工作流**——没有 lawyer 审核，AI 不能直接出回答
2. 其次是**法律合规**——免责声明、数据保护、转人工机制
3. 然后是**工程质量**——缓存、搜索优化、成本控制

如果这三个都补齐了，这个架构可以支撑上千用户量级。
